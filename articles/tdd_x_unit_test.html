TTD != Teste Unitário

Teste unitário é uma pratica onde o desenvolvedor escreve um sistema para testar o sistema que será entregue para o cliente.

TDD (Test Driven Development) é uma metologia de desenvolvimento ágil onde o programador exercita o seguinte ciclo: escreve o programa, executa o programa, e se o programa falhar ele corrige o problema. Quem pratica TDD obrigatoriamente escreve teste unitário, quem escreve teste unitário nem sempre precisa usar TDD.

Quando se implementa teste unitário sem TDD o desenvolvedor normalmente divide as tarefas de implementação e teste, primeiro ele cria o sistema que sera entregue para o cliente e depois ele desenvolve outro sistema que será responsável por testa-lo.

Quando se usa TDD não existe separação entre desenvolvimento e teste, pois para que o ciclo do TDD funcione é necessário que ambos existam.

Benefícios do TDD

Alguns problemas podem acontecer quando se decidi não usar TDD e criar a bateria de teste apenas quando o sistema final estiver pronto, como: inflexibilidade para realizar mudanças no design das classes, deixar de usar os recursos de OO (orientação ao objeto), obrigatoriedade de criar casos de teste complexos com uso excessivo de mocks e despriorização dos testes.

Flexibilidade

Se os testes indicarem quem alguma mudança de design é necessária pode ser enviável faze-la devido ao impacto, não é fácil nem rápido mudar a estrutura de classes de um sistema pronto. Com TDD esse problema é amenizado pois como o feedback dos testes é contante a necessidade de mudança seria identificado em tempo de desenvolvimento do sistema final.

Uso excessivo dos Recursos da Orientação ao Objeto

Uma linguagem orientada a objeto não força o desenvolvedor a programar orientado a objeto, é possível programar em Java proceduralmente, por exemplo. Não é incomum ver sistemas escritos Java como muitos if´s encadeados e repetição de código.

Estudos mostram uma relação diretamente proporcional entre código de qualidade e casos de tese simples, ie , se o código for bem feito ele sera fácil de ser testado.

Guiado por essa premissa o TDD foça o desenvolvedor a usar todo o leque de recursos que a orientação a objeto oferece para que os casos de teste fiquem simples.

Casos de Teste Simples

Normalmente quando usamos apenas testes unitários tendemos a não criar um design de classe voltado a teste, isso é comum pois em tempo de desenvolvimento os testes não são prioridade, eles estão planejados para uma segunda fase.

Um design de classe não orientado a teste tende a deixar os casos de testes complexos, incoesos e difícil de serem mantidos.

Garantia de Entrega dos Testes

Como os testes ficaram pra ultima etapa e o sistema já esta funcionando, é provável que o sistema seja entregue sem testes caso o tempo seja um problema.
